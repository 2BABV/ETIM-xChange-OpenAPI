## Rubber-Duck Review Findings (applied to this plan)

The following bugs were found in the original workflow snippets and have been corrected inline:

| # | Finding | Severity | Correction |
|---|---------|----------|-----------|
| 1 | `grep + awk` version extraction is fragile; fails on quoted YAML values | BLOCKING | Use `yq eval '.info.version'` |
| 2 | `sed` CHANGELOG extraction silently returns empty for the last section | BLOCKING | Use `awk` with range flag; add error if version not found |
| 3 | gh-pages accumulation snippet missing the checkout of existing `gh-pages` branch | BLOCKING | Added `actions/checkout` for gh-pages before writing new version |
| 4 | `deploy-docs.yml` and `release.yml` race to deploy Pages simultaneously | BLOCKING | Added shared `concurrency: group: pages-deploy` to both workflows |
| 5 | `${{ api_name == 'product' && 'Product' \|\| 'Trade Item' }}` is invalid GHA syntax | HIGH | Replaced with explicit `id: display` step using shell `if` |
| 6 | `deploy-docs.yml` build step ran `build:docs` but not `bundle` — missing YAML files | HIGH | Added `npm run bundle` before `build:docs` in deploy step |
| 7 | GOVERNANCE.md §7 says tag format is `vX.Y.Z`; plan uses `product/v1.2.0` | HIGH | GOVERNANCE.md update promoted to prerequisite before release workflow |
| 8 | `versions.json` not bootstrapped before first release; `index.html` would fail | HIGH | Added first-run bootstrap step to first release checklist |
| 9 | `softprops/action-gh-release@v2` is a moving target | MEDIUM | Pin to `@v2.2.0` |
| 10 | Missing CHANGELOG validation — silent failure if version missing | MEDIUM | Added `grep` guard that fails the workflow explicitly |

## Overview

Implement independent semantic versioning for each API, tag-triggered release automation, and multi-version documentation hosting via GitHub Pages. Each API is versioned, released, and documented independently.

### Key decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Version source of truth | `info.version` in each `openapi.yaml` | Single source, visible in the spec itself |
| Tag format | `product/v1.0.0`, `tradeitem/v1.1.0` | Prefixed tags allow independent releases in a mono-repo |
| Release artifacts | Bundled YAML + HTML docs | YAML for SDK generation / tooling, HTML for human consumption |
| Pages structure | `/product/latest/`, `/product/v1.0.0/` | Multi-version browsing with `latest` always pointing to newest |
| Changelog format | Per-API `CHANGELOG.md` (Keep a Changelog) | Aligns with GOVERNANCE.md §7 release notes structure |
| Version validation | Workflow fails if tag ≠ spec version | Prevents accidental version mismatches |

### Artifacts per release

| File | Purpose | Generated by |
|------|---------|-------------|
| `{api}-api.yaml` | Bundled single-file OpenAPI spec | `npm run bundle:{api}` |
| `{api}.html` | Standalone interactive API docs | `npm run build:{api}` |

## Phase 2 — Release Workflow (tag-triggered)

### 2.1 Create `.github/workflows/release.yml`

- [x] Trigger on tag push matching `product/v*` or `tradeitem/v*`
- [x] Extract API name and version from the tag (e.g., `product/v1.2.0` → `api_name=product`, `version=1.2.0`)
- [x] Validate that the tag version matches `info.version` in the corresponding `openapi.yaml` — fail the workflow if they differ
- [x] Run `npm ci`
- [x] Run `npm run lint` (full lint of all APIs as a safety net)
- [x] Run `npm run bundle:{api_name}` to generate bundled YAML
- [x] Run `npm run build:{api_name}` to generate HTML docs
- [x] Create a GitHub Release:
  - Tag: the pushed tag (e.g., `product/v1.2.0`)
  - Title: `{API Title} v{version}` (e.g., `Product API v1.2.0`)
  - Body: extracted from `CHANGELOG.md` (the section for this version)
  - Assets: attach `{api}-api.yaml` and `{api}.html`
  - Mark as pre-release if version contains `-` (e.g., `1.0.0-Preview1`)

Workflow structure:
```yaml
name: Release API

on:
  push:
    tags:
      - 'product/v*'
      - 'tradeitem/v*'

permissions:
  contents: write  # needed for creating releases

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Extract API name and version from tag
        id: extract
        run: |
          TAG="${GITHUB_REF#refs/tags/}"
          API_NAME="${TAG%/v*}"
          VERSION="${TAG#*/v}"
          echo "api_name=$API_NAME" >> "$GITHUB_OUTPUT"
          echo "version=$VERSION" >> "$GITHUB_OUTPUT"
          echo "tag=$TAG" >> "$GITHUB_OUTPUT"

      - name: Validate version matches spec
        run: |
          # Use yq (pre-installed on GitHub-hosted runners) for robust YAML parsing
          SPEC_VERSION=$(yq eval '.info.version' "openapi/apis/${{ steps.extract.outputs.api_name }}/openapi.yaml")
          TAG_VERSION="${{ steps.extract.outputs.version }}"
          if [ "$SPEC_VERSION" != "$TAG_VERSION" ]; then
            echo "::error::Tag version ($TAG_VERSION) does not match spec version ($SPEC_VERSION)"
            exit 1
          fi

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: npm

      - run: npm ci

      - name: Lint
        run: npm run lint

      - name: Bundle API spec
        run: npm run "bundle:${{ steps.extract.outputs.api_name }}"

      - name: Build HTML docs
        run: npm run "build:${{ steps.extract.outputs.api_name }}"

      - name: Extract release notes from CHANGELOG
        id: notes
        run: |
          VERSION="${{ steps.extract.outputs.version }}"
          API_NAME="${{ steps.extract.outputs.api_name }}"
          CHANGELOG="openapi/apis/$API_NAME/CHANGELOG.md"
          # Fail explicitly if version not in CHANGELOG (prevents silent empty releases)
          if ! grep -q "^## \[$VERSION\]" "$CHANGELOG"; then
            echo "::error::Version $VERSION not found in $CHANGELOG. Update the CHANGELOG before tagging."
            exit 1
          fi
          # Extract section for this version; awk handles the last section correctly
          NOTES=$(awk "/^## \[$VERSION\]/{flag=1; next} /^## \[/{flag=0} flag" "$CHANGELOG")
          echo "$NOTES" > release-notes.md

      - name: Set API display name
        id: display
        run: |
          if [ "${{ steps.extract.outputs.api_name }}" == "product" ]; then
            echo "name=Product" >> "$GITHUB_OUTPUT"
          else
            echo "name=Trade Item" >> "$GITHUB_OUTPUT"
          fi

      - name: Create GitHub Release
        uses: softprops/action-gh-release@v2.2.0
        with:
          tag_name: ${{ steps.extract.outputs.tag }}
          name: "${{ steps.display.outputs.name }} API v${{ steps.extract.outputs.version }}"
          body_path: release-notes.md
          prerelease: ${{ contains(steps.extract.outputs.version, '-') }}
          files: |
            openapi/apis/${{ steps.extract.outputs.api_name }}/generated/*
```

The current workflow deploys the repo root on every push to `main`. It needs to be enhanced to also handle versioned documentation.

- [x] Build docs into `_site/product/latest/` and `_site/tradeitem/latest/`
- [x] Upload only `_site/` (not the entire repo root) — fixes the artifact scope concern

Target Pages structure:
```
_site/
├── index.html                          ← landing page (version picker)
├── .nojekyll
├── product/
│   └── latest/
│       ├── product.html
│       └── product-api.yaml
└── tradeitem/
    └── latest/
        ├── tradeitem.html
        └── tradeitem-api.yaml
```

Updated workflow:
```yaml
# Add to both deploy-docs.yml and release.yml to prevent race conditions
concurrency:
  group: pages-deploy
  cancel-in-progress: false

- name: Build versioned site
  run: |
    mkdir -p _site/product/latest _site/tradeitem/latest

    # Bundle YAML specs first (build:docs does not generate YAML)
    npm run bundle

    # Build HTML docs
    npm run build:docs

    # Copy artifacts into versioned structure
    cp openapi/apis/product/generated/product.html _site/product/latest/
    cp openapi/apis/product/generated/product-api.yaml _site/product/latest/
    cp openapi/apis/tradeitem/generated/tradeitem.html _site/tradeitem/latest/
    cp openapi/apis/tradeitem/generated/tradeitem-api.yaml _site/tradeitem/latest/

    # Copy landing page and nojekyll
    cp index.html _site/
    cp .nojekyll _site/

- name: Upload Pages artifact
  uses: actions/upload-pages-artifact@v3
  with:
    path: _site   # upload only _site/, not entire repo root (~7.5 MB → ~1 MB)
```

> ⚠️ Add `_site/` to `.gitignore` to prevent accidentally committing the build directory.

### 3.2 Add versioned snapshots to the release workflow

When a release is tagged, the release workflow should also publish versioned docs to Pages.

- [x] In `release.yml`, after creating the GitHub Release, trigger the Pages deploy workflow with the version info
- [x] Alternatively, use a separate `deploy-versioned-docs.yml` that runs on release creation

Approach — add a Pages deploy step to `release.yml`:
```yaml
      - name: Checkout gh-pages branch (accumulate existing versions)
        uses: actions/checkout@v4
        with:
          ref: gh-pages
          path: gh-pages-content
        # On first run, the branch won't exist. Create it:
        # git checkout --orphan gh-pages && git rm -rf . && git commit --allow-empty -m "init"
        # Push once manually before the first release: git push origin gh-pages

      - name: Create versioned Pages directory
        run: |
          API="${{ steps.extract.outputs.api_name }}"
          VER="v${{ steps.extract.outputs.version }}"

          # Preserve existing versions from gh-pages
          mkdir -p _site
          cp -r gh-pages-content/* _site/ 2>/dev/null || true

          # Add new versioned snapshot
          mkdir -p "_site/$API/$VER" "_site/$API/latest"
          cp "openapi/apis/$API/generated/"* "_site/$API/$VER/"
          cp "openapi/apis/$API/generated/"* "_site/$API/latest/"
```

> ⚠️ **First-run prerequisite**: The `gh-pages` branch must exist before the first tag push. Create it manually once:
> ```bash
> git checkout --orphan gh-pages
> git rm -rf .
> git commit --allow-empty -m "chore: init gh-pages branch"
> git push origin gh-pages
> git checkout main
> ```
